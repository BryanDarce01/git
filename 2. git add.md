# `git add`

Sirve para preparar archivos antes de guardarlos en un commit.

Dicho en palabras simples:

- `git add` le dice a **Git** qué archivos se van a incluir en el próximo commit.

# ⚙️ Cómo funciona internamente

**Git** tiene 3 áreas principales donde maneja los archivos:


- 1️⃣ **Working Directory**  → La carpeta actual (archivos reales)

- 2️⃣ **Staging Area**       → Zona de preparación (con `git add`)

- 3️⃣ **Repository** (Commit)→ Historial guardado (con `git commit`)


# Flujo típico:

- 1. Se modifican o crean archivos → están en el **Working Directory**.

- 2. Se ejecuta `git add archivo.js` → **Git** los pasa al **Staging Area**.

- 3. Luego se hace `git commit` → **Git** los guarda en el **Repository (historial)**.
    - Esto toma todos los archivos en staging (agregados con `git add`)
y los guarda de forma permanente en el historial del repositorio.

# 🧱 Estructura interna de Git (modelo mental)

```
┌─────────────────────────────┐
│      Repositorio (Commits)  │   ← HEAD apunta aquí
└───────────────┬─────────────┘
                │
                ▼
┌─────────────────────────────┐
│     Staging Area (Index)    │   ← lo que se incluirá en el próximo commit
└───────────────┬─────────────┘
                │
                ▼
┌─────────────────────────────┐
│   Working Directory (WD)    │   ← tus archivos editados en el PC
└─────────────────────────────┘
```

# Más comandos

# 1. `git add <file>`

Agrega un archivo específico al **staging**.

Ejemplo:

``` bash
git add app.js
git add src/utils.py
```

# 2. `git add .`

Agrega todos los cambios del directorio actual hacia abajo:

- nuevos

- modificados

- borrados

**Es el más usado en la práctica.**

# 3. `git add -A (o --all)`

Agrega absolutamente todos los cambios de **TODO** el repo, incluso **fuera** del cwd.

# `4. git add -u (o --update)`

Agrega **SOLO** archivos modificados o borrados, pero **NO** archivos nuevos.

Muy útil en pipelines.

# 5. `git add -p (o --patch)`

Modo interactivo para agregar **solo partes específicas** de un archivo.

Permite revisar cada hunk:

Git mostrará cada bloque de cambios:

``` bash
Stage this hunk [y,n,q,a,d,s,e,?]?
```

# 6. `git add -i (interactive)`

Abre un menú interactivo más completo (como una UI en terminal):

Permite:

- seleccionar archivos

- revisar parches

- ver diferencias

- quitar del stage

# 7. `git add --renormalize`

Re-normaliza archivos según `.gitattributes`.
Muy usado en proyectos grandes con equipos que usan diferentes OS.

``` bash
git add --renormalize .
```

**Corrige final de línea (CRLF/LF), codificación, etc.**

# 8. `git add -N (o --intent-to-add)`

Marca un archivo como **“existente en el stage”** sin realmente añadir contenido.

Sirve para:

- permitir `git diff --cached` antes del primer add

- preparar archivos nuevos para revisión

Ejemplo:

``` bash
git add -N secret.env
```

#  9. `git add --force (o -f)`

Forza agregar archivos que están ignorados por `.gitignore`.

Ejemplo:

``` bash
git add -f logs/important.log
```

# 10. `git add --ignore-errors`

Continúa aunque algunos archivos fallen al agregar.

``` bash
git add --all --ignore-errors
```

# 📚 RESUMEN
``` bash

Command               | Qué agrega
--------------------------------------------------------
git add <file>        | Un archivo específico
git add .             | Todo en el directorio actual
git add -A / --all    | Todo en el repo completo
git add -u            | Solo modificados y borrados (no nuevos)
git add -p            | Partes de archivos (patch mode)
git add -i            | UI interactiva en terminal
git add -N            | "Intent-to-add" (metadatos sin contenido)
git add -f            | Forzar agregar ignorados
git add --renormalize | Renormalizar según .gitattributes
git add --ignore-errors | No detenerse en errores

```

# 🎯 Qué NO hace git add

- ❌ No hace commit
- ❌ No sube al remoto
- ❌ No crea ramas
- ❌ No guarda historial
- ❌ No revierte nada

**Solo prepara la fotografía del siguiente commit.**
